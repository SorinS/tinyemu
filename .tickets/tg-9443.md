# TCP SYN-ACK packets not delivered to guest despite successful VirtIO write


## Notes

**2026-01-23T15:29:22Z**

## Summary

TCP SYN-ACK packets are successfully written to the VirtIO network RX queue but are not processed by the guest. This causes TCP connections from guest to host to fail with "Connection timed out" while DHCP (UDP) and ICMP ping work correctly.

## Symptoms

- `TestLinuxBootTCPHostServer` fails - guest cannot establish TCP connection to host
- `TestLinuxBootNetwork` passes - DHCP and ping work correctly
- Guest sees ARP replies (required for TCP SYN to work) but not TCP SYN-ACK

## Investigation Findings (2026-01-23)

### What Works
- ARP request/reply: Guest sends ARP, slirp sends ARP reply, guest receives it
- DHCP (UDP): Guest runs udhcpc, slirp responds, guest receives DHCP Offer/ACK
- ICMP ping: Guest pings gateway, slirp responds, guest receives echo reply

### What Doesn't Work
- TCP: Guest sends SYN, slirp sends SYN-ACK, guest NEVER receives SYN-ACK
- Guest retransmits SYN repeatedly until timeout

### VirtIO Layer - VERIFIED WORKING
- **14 packets written to VirtIO queue, 0 dropped**
- Breakdown: 2 ARP replies + 12 TCP SYN-ACK retransmits
- All packets successfully written to different descriptors (0-13)
- **Fixed**: Used ring entry write order now matches C TinyEMU (write entry data BEFORE incrementing index)
- Used ring index increments correctly (verified: usedIdx matches expected values)

### Interrupt Delivery - VERIFIED WORKING
- PLIC raises IRQ 3 for each packet (level=1 observed in traces)
- Guest DOES claim the interrupt (served bit gets set)
- Guest DOES acknowledge VirtIO interrupt (InterruptACK handler called)
- Guest DOES complete PLIC interrupt (complete handler called with served bit cleared)

### THE KEY MYSTERY
Despite all the above working correctly:
1. VirtIO used ring is updated correctly ✓
2. IRQ is raised ✓
3. Guest claims interrupt ✓
4. Guest acknowledges VirtIO ✓
5. Guest completes PLIC ✓

...the guest's network stack does NOT receive the TCP SYN-ACK packet. The virtio-net driver's NAPI handler appears to run (it acknowledges the interrupt) but either:
- Does NOT read from the used ring, OR
- Reads from used ring but packet is corrupted/invalid, OR
- Some Linux kernel TCP-specific handling rejects the packet

### Available Ring State
- Guest provides 16 descriptors initially (availIdx=16)
- Descriptors are NOT being recycled (availIdx stays at 16)
- This confirms guest is NOT processing packets from used ring

### Packet Analysis
TCP SYN-ACK packets are well-formed:
- Ethernet header: Correct destination MAC (02:00:00:00:00:01), correct source MAC
- IP header: Valid checksum, correct addresses (10.0.2.2 -> 10.0.2.15)
- TCP header: Correct SYN+ACK flags, valid sequence/ack numbers, MSS option
- Packet size: 58 bytes (14 eth + 20 IP + 24 TCP with options)

## Root Cause Hypotheses (Prioritized)

1. **VirtIO used ring entry format issue**: The used ring entry (id, len) might have wrong format or values that confuse the Linux driver

2. **Descriptor flags mismatch**: The RX descriptors might have flags that Linux expects but we're not respecting

3. **VirtIO feature negotiation**: Missing required feature bits that Linux virtio-net driver expects for proper RX handling

4. **Memory barrier requirement**: Even in single-threaded emulation, Linux may require specific memory ordering we're not providing

5. **NAPI poll scheduling**: Linux schedules NAPI but it doesn't run because of missing softirq trigger

## Reproduction

```bash
go test -v -run "TestLinuxBootTCPHostServer" ./machine/ -timeout 180s
```

### TCP Data Exchange Now Works!

Major breakthrough: TCP bidirectional data exchange is now functional:
- Guest successfully sends "HELLO_FROM_GUEST\n" to host server
- Host server receives the data and sends "RESPONSE_FROM_HOST\n" back  
- Packets are correctly written to VirtIO and delivered to guest

### Fix Applied: TCP Checksum Pseudo-header

The issue was in TCP checksum calculation. The C code uses a union where:
- ti_mbuf (bytes 0-7) overlays IP header fields
- ti_x1 (byte 8) overlays ip_ttl

Before checksum calculation, C sets `ti_mbuf = NULL` and `ti_x1 = 0`. 
The Go code was not fully clearing these bytes. Fixed by explicitly 
clearing bytes 0-8 to 0 in marshalTemplate() and marshalTCPIPHdrTo().

The actual IP header values are restored by ipOutput() after checksum.

### Remaining Issue: TCP Connection Close

The test still times out because TCP FIN handling doesn't complete:
- Server sends FIN+ACK after sending response data
- Guest doesn't respond with ACK to close connection  
- FIN+ACK packets are retransmitted until timeout
- nc command never exits, so 4th shell prompt never appears

The data path works; the close path needs investigation. Possible causes:
1. Guest virtio-net driver not processing FIN packets correctly
2. Guest TCP stack issue with half-close handling
3. Timing/scheduling issue in emulator
