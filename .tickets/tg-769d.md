# Implement getDnsAddr function for DNS forwarding

The tcpFConnect function (and similar code in socket.go and ip_icmp.go) needs to call get_dns_addr to resolve the system's configured DNS server when connecting to the emulated nameserver.

Currently, when the guest tries to connect to the virtual nameserver (vnetwork.nameserver_addr), the code should call get_dns_addr to find the host's actual DNS server. Instead, it falls back to LoopbackAddr unconditionally.

## C Reference

The C implementation in slirp/slirp.c:48 (Windows) and slirp/slirp.c:102 (Unix) parses /etc/resolv.conf to extract the nameserver address.

## Files Affected

- slirp/tcp_input.go:1559-1562 (tcpFConnect)
- Likely also socket.go and ip_icmp.go where similar patterns exist

## Expected Behavior

When the guest connects to the virtual nameserver, forward the connection to the host's actual DNS server (from /etc/resolv.conf on Unix, or Windows API on Windows).

## Current Behavior

Always uses LoopbackAddr (127.0.0.1), which only works if a DNS server is running locally.
